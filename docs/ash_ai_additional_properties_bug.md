# AshAi Anthropic Compatibility Bugs

## Summary

When AshAi tool definitions are used with the Anthropic API (via `LangChain.ChatModels.ChatAnthropic`), API calls fail with:

```
tools.N.custom: For 'object' type, 'additionalProperties' must be explicitly set to false
```

The Anthropic API requires every JSON Schema object to include `"additionalProperties": false`. Several nested objects in the tool schemas generated by `AshAi` are missing this field.

## How to Reproduce

Define any AshAi tools exposing a `:read` action on a resource with public attributes, and call them via a prompt-backed action using `ChatAnthropic`:

```elixir
# Domain with tools
defmodule MyApp.Blog do
  use Ash.Domain, otp_app: :my_app, extensions: [AshAi]

  tools do
    tool :read_posts, MyApp.Blog.Post, :read do
      load [:author]
    end
  end

  resources do
    resource MyApp.Blog.Post do
      define :generate_summary, action: :generate_summary, args: [:post_id]
    end
  end
end

# Resource with a prompt-backed action using tools
defmodule MyApp.Blog.Post do
  use Ash.Resource, otp_app: :my_app, domain: MyApp.Blog, data_layer: AshPostgres.DataLayer

  require AshAi.Actions

  actions do
    defaults [:read]

    action :generate_summary, :string do
      argument :post_id, :integer, allow_nil?: false

      run AshAi.Actions.prompt(
            fn _input, _context ->
              LangChain.ChatModels.ChatAnthropic.new!(%{
                model: "claude-sonnet-4-20250514",
                api_key: System.get_env("ANTHROPIC_API_KEY")
              })
            end,
            tools: [:read_posts],
            prompt: "Summarize post <%= @input.arguments.post_id %>"
          )
    end
  end

  attributes do
    integer_primary_key :id
    attribute :title, :string, public?: true
    attribute :body, :string, public?: true
    timestamps()
  end
end
```

Calling `MyApp.Blog.generate_summary!(1, actor: some_user)` produces:

```
[warning] Unexpected and unhandled API response!
{:ok, %Req.Response{status: 400, body: %{
  "error" => %{
    "message" => "tools.1.custom: For 'object' type, 'additionalProperties' must be explicitly set to false",
    "type" => "invalid_request_error"
  }
}}}
```

This works fine with `ChatOpenAI` because OpenAI does not enforce `additionalProperties` on tool schemas.

## Root Cause

The tool parameter schemas are built in `lib/ash_ai/tools.ex` function `add_action_specific_properties/4`. For `:read` actions, it generates `filter`, `sort`, and `result_type` sub-schemas. Several of these nested `type: :object` maps are missing `additionalProperties: false`.

There is also one location in `lib/ash_ai/open_api.ex` where a fallback `%{type: :object}` is returned for unconstrained `Ash.Type.Map` attributes without the field.

## Affected Locations

### File: `lib/ash_ai/tools.ex`

#### 1. `filter` object (line ~513)

The `filter` property is a nested object but lacks `additionalProperties`:

```elixir
filter: %{
  type: :object,
  description: "Filter results",
  properties: ...
  # MISSING: additionalProperties: false
}
```

**Fix:** Add `additionalProperties: false` to the map.

#### 2. `sort.items` object (line ~585)

Each sort item is an object with `field` and `direction` properties, but lacks `additionalProperties`:

```elixir
sort: %{
  type: :array,
  items: %{
    type: :object,
    properties: %{
      field: %{...},
      direction: %{...}
    }
    # MISSING: additionalProperties: false
  }
}
```

**Fix:** Add `additionalProperties: false` to the `items` map. Note that `add_input_for_fields/2` may add an `input_for_fields` key to this map, so `additionalProperties: false` should be added after that call, or the `add_input_for_fields` function should ensure it's present.

#### 3. `result_type.oneOf[1]` aggregate object (line ~542)

The second variant in the `oneOf` for `result_type` is an object with `aggregate` and `field` properties, but lacks both `type: :object` and `additionalProperties`:

```elixir
%{
  properties: %{
    aggregate: %{
      type: :string,
      description: "The aggregate function to use",
      enum: [:max, :min, :sum, :avg, :count]
    },
    field: %{
      type: :string,
      description: "The field to aggregate",
      enum: [...]
    }
  }
  # MISSING: type: :object
  # MISSING: additionalProperties: false
}
```

**Fix:** Add `type: :object` and `additionalProperties: false`.

#### 4. Typo: `additonalProperties` in `input_for_fields` wrapper (line ~649)

The `input_for_fields` wrapper object has a typo — `additonalProperties` instead of `additionalProperties`:

```elixir
input_for_fields =
  %{
    type: :object,
    additonalProperties: false,  # <-- TYPO: missing 'i' in 'additional'
    properties: ...
  }
```

**Fix:** Rename to `additionalProperties: false`.

### File: `lib/ash_ai/open_api.ex`

#### 5. Unconstrained `Ash.Type.Map` in `resource_write_attribute_type/3` (line ~81)

When a `Map` type has no field constraints, it falls through to a bare `%{type: :object}`:

```elixir
def resource_write_attribute_type(
      %{type: Ash.Type.Map, constraints: constraints} = attr,
      resource,
      action_type
    ) do
  if constraints[:fields] && constraints[:fields] != [] do
    %{
      type: :object,
      additionalProperties: false,  # present here
      ...
    }
  else
    %{type: :object}  # <-- MISSING additionalProperties
  end
end
```

**Fix:** Change to `%{type: :object, additionalProperties: false}`.

#### 6. Unconstrained `Ash.Type.Map` in `resource_attribute_type/2` (line ~507)

Same pattern as above, in the read-side function:

```elixir
defp resource_attribute_type(
       %{type: Ash.Type.Map, constraints: constraints} = attr,
       resource
     ) do
  if constraints[:fields] && constraints[:fields] != [] do
    %{type: :object, additionalProperties: false, ...}  # present here
  else
    %{type: :object}  # <-- MISSING additionalProperties
  end
end
```

**Fix:** Change to `%{type: :object, additionalProperties: false}`.

## Complete Fix Summary

| File | Line (approx.) | Location | Issue |
|------|----------------|----------|-------|
| `tools.ex` | 513 | `filter` object | Missing `additionalProperties: false` |
| `tools.ex` | 585 | `sort.items` object | Missing `additionalProperties: false` |
| `tools.ex` | 542 | `result_type.oneOf[1]` | Missing `type: :object` and `additionalProperties: false` |
| `tools.ex` | 649 | `input_for_fields` wrapper | Typo: `additonalProperties` should be `additionalProperties` |
| `open_api.ex` | 81 | `resource_write_attribute_type` Map fallback | Missing `additionalProperties: false` |
| `open_api.ex` | 507 | `resource_attribute_type` Map fallback | Missing `additionalProperties: false` |

## Notes

- The top-level tool schema and the `input` sub-object in `tools.ex` already have `additionalProperties: false` set correctly (lines 489, 500). The issue is only in the nested objects added by `add_action_specific_properties/4` for `:read` actions and in the Map type fallbacks.
- The `LangChain.ChatModels.ChatAnthropic` adapter in LangChain (`deps/langchain/lib/chat_models/chat_anthropic.ex`) does add `additionalProperties: false` to the top-level `parameters_schema` via `Map.put_new/3` (line ~2097), but it does not recursively fix nested objects — nor should it, since that's the schema generator's responsibility.
- The `completion_tool.ex` adapter in AshAi (used for Anthropic) correctly sets `additionalProperties: false` on its own `complete_request` tool schema.
- OpenAI's API does not enforce this constraint, which is why the bug only manifests with Anthropic.
- These fixes are all additive (adding a missing key to a map) and should not affect behavior with other LLM providers, since `additionalProperties: false` is valid JSON Schema regardless of provider.

# Bug 2: CompletionTool Adapter Sets `strict: true` Unconditionally

## Summary

The `CompletionTool` adapter (used by default for Anthropic) creates its `complete_request` tool with `strict: true`. When LangChain serializes this for the Anthropic API, it sends `"strict": true` in the tool definition. Anthropic does not support strict tools and rejects the request:

```
'claude-sonnet-4-20250514' does not support strict tools.
```

This means **any** prompt-backed action using `ChatAnthropic` fails, even without AshAi tools — the `complete_request` tool alone triggers the error.

## How to Reproduce

Any prompt-backed action with `ChatAnthropic` and the default adapter:

```elixir
action :analyze, :string do
  argument :text, :string, allow_nil?: false

  run AshAi.Actions.prompt(
        LangChain.ChatModels.ChatAnthropic.new!(%{
          model: "claude-sonnet-4-20250514",
          api_key: System.get_env("ANTHROPIC_API_KEY")
        }),
        tools: false
      )
end
```

This fails because the adapter selection logic picks `CompletionTool` for `ChatAnthropic`, and `CompletionTool` creates:

```elixir
LangChain.Function.new!(%{
  name: "complete_request",
  ...
  strict: true,   # <-- causes the error
  ...
})
```

LangChain's `ChatAnthropic.for_api/3` then includes `"strict": true` in the API request (line ~1753 of `chat_anthropic.ex`), which Anthropic rejects.

## Root Cause

**File:** `lib/ash_ai/actions/prompt/adapter/completion_tool.ex`, line 53

```elixir
completion_tool =
  LangChain.Function.new!(%{
    name: "complete_request",
    description: description,
    parameters_schema: %{...},
    strict: true,       # <-- This field
    async: false,
    function: fn arguments, _context -> ... end
  })
```

The `strict` flag is an OpenAI feature for structured outputs. It has no meaning for Anthropic and actively causes an error.

## Fix

The `strict` flag should only be set when the LLM supports it. The adapter has access to the LLM via `data.llm`, so it can check the type:

```elixir
completion_tool =
  LangChain.Function.new!(%{
    name: "complete_request",
    description: description,
    parameters_schema: %{...},
    strict: match?(%LangChain.ChatModels.ChatOpenAI{}, data.llm),
    async: false,
    function: fn arguments, _context -> ... end
  })
```

Alternatively, `strict` could be omitted entirely (set to `nil` or `false`) and only set to `true` when the LLM is known to support it. LangChain already handles this correctly — it only sends `"strict"` in the API payload when `fun.strict` is truthy (see `chat_anthropic.ex` line ~1753).

## Workaround

Force the `RequestJson` adapter, which does not use tool calling for structured outputs:

```elixir
run AshAi.Actions.prompt(
      fn _input, _context -> Xo.Games.LLM.build() end,
      adapter: AshAi.Actions.Prompt.Adapter.RequestJson,
      tools: false
    )
```
